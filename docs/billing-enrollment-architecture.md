# QAS Billing and Enrollment Architecture

## Purpose

QAS needs a consistent billing model for trial lessons, full-term students, pay-as-you-go/ad hoc bookings, holiday programs, and non-course product orders.

The core rule is:

- Learning participation is represented by `Enrollment`.
- Non-course purchases are represented by `Order`.
- Invoices are generated by a billing service from either an `Enrollment` or an `Order`.
- `Inquiry` remains a CRM/lead workflow and should not directly own final billing.

## Current State

`Course` and `Enrollment` currently exist as manually maintained DocTypes in production. Their fields are visible in the Desk UI, but they are not yet controlled by `qas_custom` JSON files.

Current `Course` fields include:

- `course_name`
- `status`
- `duration`
- `max_age`
- `min_age`
- `full_term_fee`
- `pay_as_you_go_fee`
- `total_session_per_term`
- `term_session_fee`
- `is_makeup_course`
- `accepted_makeup_course`

Current `Enrollment` fields include:

- `student`
- `term`
- `course`
- `weekly_timeslot`
- `enrollment_type`
- `status`
- `trial_session_date`
- `enrollment_date`
- `pay_as_you_go_date`

Existing code references `Enrollment` for active course membership and term rollover, but no existing code implements pro-rata invoice generation from `Enrollment`.

## Target Concepts

### Inquiry

`Inquiry` represents a lead and trial/school-visit workflow.

It owns:

- family/contact information
- trial booking and attendance status
- follow-up notes
- conversion state
- inactive state and inactive reason

It should not directly calculate tuition or create final invoices.

When a trial lead converts, `Inquiry` triggers creation of a full-term `Enrollment`. The resulting draft invoice is attached back to the inquiry for traceability.

### Enrollment

`Enrollment` represents a student's participation in a learning product.

Enrollment types:

- `Trial`
- `Full-Term`
- `Pay-as-you-go`
- `Holiday Program`

Common fields:

- `student`
- `parent`
- `customer`
- `enrollment_type`
- `status`
- `course`
- `term`
- `weekly_timeslot`
- `start_course_session`
- `start_date`
- `end_date`
- `session_count`
- `billing_status`
- `invoice`
- `source_doctype`
- `source_name`

`Full-Term` enrollments use `term`, `weekly_timeslot`, and `start_course_session`.

`Pay-as-you-go` enrollments use one selected `Course Sessions` record.

`Trial` enrollments use one selected trial `Course Sessions` record.

`Holiday Program` enrollments use a holiday program offering instead of a weekly timeslot.

### Holiday Program Offering

Holiday programs are not regular weekly courses. They may be a single full-day class or a consecutive multi-day program.

They should not be forced into `Weekly Timeslot`.

Introduce a new offering concept:

- `Program Offering`
- or more specifically `Holiday Program Offering`

Suggested fields:

- `program_name`
- `offering_type = Holiday Program`
- `course`
- `campus`
- `start_date`
- `end_date`
- `price`
- `status`
- child table of program sessions/days

Holiday program invoice amount comes from the offering price, not from weekly term pro-rata logic.

### Order

`Order` represents non-course purchases.

Examples:

- art materials
- supply packs
- merchandise
- non-course services

Orders do not create enrollment records.

Suggested fields:

- `parent`
- `customer`
- `student` optional
- `status`
- `invoice`
- child table `Order Item`

Order items should map cleanly to ERPNext `Sales Invoice Item` rows.

## Invoice Ownership

All generated invoices should be ERPNext `Sales Invoice` records.

Generation paths:

- `Enrollment -> Draft Sales Invoice`
- `Order -> Draft Sales Invoice`

Draft invoices are created for School Admin review. Parent Portal should continue showing only submitted invoices unless a later requirement asks to expose draft invoices.

QAS should not fork or replace ERPNext `Sales Invoice` / `Sales Invoice Item`. The standard ERPNext invoice remains the accounting record, and QAS-specific context is added through custom fields.

Invoice header is family/customer-level:

- `customer`
- `parent`
- `primary_student` only as a convenience/list hint
- `student_summary`
- `qas_invoice_type`: Course, Store Credit Top-up, Holiday Program, Material Order, Other
- `source_doctype`
- `source_document`
- `billing_note`

The authoritative student/course context belongs on each `Sales Invoice Item` row:

- `qas_line_type`: Course Fee, Adhoc Credit, Holiday Program, Material, Other
- `student`
- `student_code`
- `enrollment`
- `course`
- `term`
- `course_session`
- `session_count`

This supports one parent receiving a single draft invoice containing fees for multiple children. The item description must also be human-readable, for example:

```text
Isabella02 - Anime Art - Intermediate - Term 3 prorata, 5 sessions
```

Material orders can use the same ERPNext invoice object, but their item rows may leave student/enrollment/course fields blank and use `qas_line_type = Material`. Course store credit rules apply only to course-related invoice types/items, not to material/order invoices.

## Full-Term Conversion Flow

Campus Admin can convert a completed trial lead.

Flow:

1. Campus Admin opens `Post-trial Follow-up`.
2. Campus Admin chooses `Convert`.
3. UI asks for a formal start date.
4. System lists course sessions on that date.
5. Campus Admin chooses the exact start session.
6. The selected session determines:
   - `weekly_timeslot`
   - `course`
   - `term`
   - campus/class time
7. System creates a `Full-Term` enrollment.
8. System calculates pro-rata tuition.
9. System creates a draft `Sales Invoice`.
10. Inquiry is updated:
   - `status = Converted`
   - `converted_enrollment = enrollment`
   - invoice link is stored for audit/review

The selected formal course does not need to match the trial course.

## Full-Term Pro-Rata Calculation

Course fee source:

- `Course.full_term_fee`

Session divisor:

- `Course.total_session_per_term`
- default expectation is `9`
- avoid hard-coding `9` so future terms can differ

Formula:

```text
per_session_fee = Course.full_term_fee / Course.total_session_per_term
remaining_sessions = count of Course Sessions for the same weekly_timeslot and term
                     where session_date >= selected_start_session.session_date
                     and status != Cancelled
invoice_amount = per_session_fee * remaining_sessions
```

The selected start session is included in `remaining_sessions`.

The invoice should be created in draft mode.

## Pay-As-You-Go / Ad Hoc Flow

Ad hoc students should also be represented by `Enrollment`.

Flow:

1. Parent or staff selects a specific course session.
2. System creates a `Pay-as-you-go` enrollment for that session.
3. Amount uses `Course.pay_as_you_go_fee`.
4. Billing service creates either:
   - a draft Sales Invoice, or
   - a balance hold/charge if QAS keeps the account-balance workflow

The preferred long-term model is still `Enrollment` as the source record.

## Trial Flow

Trial lessons should be represented by a `Trial` enrollment.

Flow:

1. Inquiry books a trial session.
2. System creates or links a `Trial` enrollment.
3. If trial is paid, billing service creates a draft Sales Invoice from the trial enrollment.
4. If trial is free, no invoice is created.

This keeps attendance, billing, and learning participation consistent.

## Inactive Flow

`Inactive` is set by Campus Admin during post-trial follow-up. School Admin can also update it.

Rules:

- `inactive_reason` is required when setting `Inactive`.
- `Inactive` means the lead/enrollment is no longer being pursued.
- `Cancelled` means a specific trial/session was cancelled and is not the same as inactive.

Suggested reason examples:

- parent not interested
- price concern
- schedule not suitable
- selected another provider
- unreachable
- duplicate inquiry
- other

## Required DocType Migration

Move these production-manual DocTypes into code-controlled JSON under `qas_custom`:

- `Course`
- `Enrollment`

Add any required custom child DocTypes:

- accepted makeup course child table if not already code-controlled
- `Order`
- `Order Item`
- `Holiday Program Offering`
- `Holiday Program Session`

Migration must preserve production data.

Recommended migration steps:

1. Export or reconstruct current `Course` and `Enrollment` DocType JSON.
2. Add them to `qas_custom/qas_custom/doctype`.
3. Add patches that safely add missing fields when tables already exist.
4. Run migration on staging/production.
5. Verify existing records remain readable.

## Services To Add

### Billing Service

Create `qas_custom/services/billing.py`.

Responsibilities:

- build draft Sales Invoice from Enrollment
- build draft Sales Invoice from Order
- resolve customer from parent/student
- resolve invoice item and income account
- calculate full-term pro-rata amount
- return invoice metadata

### Enrollment Service

Create `qas_custom/services/enrollment.py`.

Responsibilities:

- create `Trial` enrollment
- create `Full-Term` enrollment from selected start session
- create `Pay-as-you-go` enrollment
- create `Holiday Program` enrollment
- validate duplicate active enrollment
- link enrollment to source records

### Inquiry Conversion Service

Extend inquiry/campus admin services with:

- get conversion session candidates by date
- convert inquiry to full-term enrollment
- mark inquiry inactive with reason

## API Surface

Campus Admin API:

- `campus_admin_get_conversion_sessions(inquiry, start_date, course=None, campus=None)`
- `campus_admin_convert_inquiry(inquiry, course_session)`
- `campus_admin_mark_inquiry_inactive(inquiry, inactive_reason)`

School Admin/API:

- can edit Enrollment and draft Sales Invoice through Desk
- can submit invoices after review

Order API can be added later:

- create order
- update order items
- create draft invoice from order

## Parent Portal Behavior

Parent Portal should keep showing only submitted invoices for now.

Draft invoices created from conversion are for School Admin review and should not appear to parents until submitted.

## Implementation Phases

### Phase 1: Code-Control Core DocTypes

- Move `Course` and `Enrollment` into code.
- Add migration safeguards.
- Validate production data shape.

### Phase 2: Full-Term Conversion

- Add enrollment service.
- Add billing service.
- Add Campus Admin convert UI.
- Create full-term enrollment and draft invoice from selected start session.
- Add inactive workflow with required reason.

### Phase 3: Trial and Ad Hoc Unification

- Make trial booking create/link `Trial` enrollment.
- Make ad hoc booking create/link `Pay-as-you-go` enrollment.
- Preserve existing ad hoc balance-hold behavior until deliberately replaced.

### Phase 4: Holiday Programs

- Add holiday program offering model.
- Add holiday program enrollment.
- Add draft invoice generation from offering price.

### Phase 5: Orders

- Add Order and Order Item.
- Add draft invoice generation from order.
